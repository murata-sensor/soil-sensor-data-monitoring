# 01. Google 側セットアップ（Spreadsheet / サービスアカウント / OAuth）

> **所要時間**: 約 20〜30 分
> **前提**: Google アカウントを持っていること、GitHub リポジトリが作成済みであること

---

## 0. アクセス制御の考え方（先に理解してから 1. へ進む）

このシステムでは **2 段階** の制御があります。両方そろって初めて閲覧できます。

### 段階 1: 「そのアカウントはダッシュボードにログインできるか？」

`users` シートで管理する。`enabled=TRUE` なら **ログインだけは** できる。
ただし `enabled=TRUE` でも **どのスプレッドシートが見えるかは別問題**。

### 段階 2: 「ログインしたアカウントは、どのスプレッドシートを見られるか？」

`acl` シートで **1 件ずつ** 許可する。`acl` に行が無いユーザーは **ログインはできるけど画面が空っぽ**。

**例**: 3 つの圃場（src-001 / src-002 / src-003）に、3 人のユーザーを割り当てる場合

`users` シート（ログイン可否）:

| email | role | enabled |
| --- | --- | --- |
| `alice@example.com` | `admin` | `TRUE` |
| `bob@example.com`   | `viewer` | `TRUE` |
| `carol@example.com` | `viewer` | `TRUE` |

`acl` シート（誰がどの圃場を見られるか）:

| email | sourceId | permission |
| --- | --- | --- |
| `bob@example.com`   | `src-001` | `read` |
| `carol@example.com` | `src-002` | `read` |
| `carol@example.com` | `src-003` | `read` |

実際にログインしたときの見え方:

- **alice**（admin）→ src-001, src-002, src-003 の **全部** が見える（admin は acl をスキップ）
- **bob** → src-001 だけ見える
- **carol** → src-002 と src-003 が見える（src-001 は見えない）

> 「`enabled=TRUE` にしたから全部見える」ということではない、というのが重要なポイント。
> 圃場を増やしたら `acl` シートに **明示的にそのユーザーの行を追加** する必要があります。

> **おまけ条件**: `acl` で許可していても、データ用スプレッドシートが `accessMode=direct` の場合は
> **そのユーザーの Google アカウントを直接「閲覧者」共有しておく必要があります**（後述）。
> `accessMode=proxy` にすればこの Drive 共有を省略できます。

## 1. Registry スプレッドシートを作成する（プロジェクト所有）

「Registry」は **制御専用** のスプレッドシート。実データ（センサ測定値）はここには入りません。
「どのスプレッドシート（=データ源）があるか」「誰がログインできるか」「誰がどれを見られるか」だけを書きます。

> **作るのは 1 個だけ**。後からデータ源が増えても、シートを増やすのではなく `sources` の行を追加していきます。

### 1-1. 空のスプレッドシートを 1 枚作る

1. Web ブラウザで <https://sheets.google.com> を開く
2. 画面左上、色付きの「**＋ 空白**」（または 「Blank」）の四角をクリック
3. 新しい空のスプレッドシートが開く。左上に **「無題のスプレッドシート」** と書かれている部分をマウスでクリックする
4. 名前を `SoilSensorRegistry` と入力して `Enter` キーを押す
   - 地名や顧客名は付けないこと（リポジトリにスクリーンショットを貼る可能性があるため）

### 1-2. 6 つのシート（タブ）を作って名前を付ける

スプレッドシート画面の **下部** に「シート1」というタブが 1 つだけある状態。これを 6 つにする。

#### (a) 「シート1」を `sources` にリネーム

1. 画面下部の「**シート1**」タブを **右クリック**
2. メニューから「**名前を変更**」を選ぶ
3. `sources` と入力 → `Enter`

#### (b) タブを 5 枚追加

1. 画面左下の **「＋」ボタン**（タブの左隣）を 5 回クリックして、タブを「シート2 / 3 / 4 / 5 / 6」と追加
2. それぞれ右クリック → 「名前を変更」で以下のように名前を付ける:
   - `users`
   - `acl`
   - `theme`
   - `events`
   - `layouts`

最終的に **下部のタブ並びが `sources | users | acl | theme | events | layouts`** の 6 つになっていれば OK（順番は自由）。

### 1-3. 各シートの 1 行目（ヘッダ）を入力する

各タブをクリックして、**1 行目（A1 から右へ）** に下表のヘッダを 1 セルずつ入力します。

> **時短ワザ**: 下表の「ヘッダ（カンマ区切り）」列をマウスでドラッグして選択 → `Ctrl+C` でコピー →
> シートの A1 セルに `Ctrl+V` で貼り付ける → メニュー「**データ → テキストを列に分割**」→ 区切り文字「カンマ」を選ぶ。
> これで 1 行に展開される。

| シート名 | A1 から順に入力するヘッダ（カンマ区切り） |
|----------|---------------------------------------------|
| `sources` | `sourceId,displayName,schemaType,spreadsheetId,sheetName,headerRow,siteId,tz,accessMode,enabled,notes` |
| `users`   | `email,role,enabled` |
| `acl`     | `email,sourceId,permission` |
| `theme`   | `themeId,json` |
| `events`  | `date,sourceId,label,color,deviceId` |
| `layouts` | `sourceId,config` |

各列の意味（後で値を入れるとき用）:

**`sources` シート**

| 列 | 値の例 | 意味 |
| --- | --- | --- |
| `sourceId` | `src-001` | 内部 ID。英数字＋ハイフンで自分が分かりやすい名前 |
| `displayName` | `A 棟 M5Stack` | ダッシュボードのドロップダウンに出る表示名 |
| `schemaType` | `m5stack` / `mechatrax` / `remote-ftp` | データ形式 |
| `spreadsheetId` | `1abcDEF...xyz` | 実データスプレッドシートの ID（URL から取得、後述） |
| `sheetName` | `soil_sensor` | 実データが入っているタブ名 |
| `headerRow` | `1` | 通常 `1`。ヘッダが何行目にあるか |
| `siteId` | `site-a` | 圃場識別子（自由） |
| `tz` | `Asia/Tokyo` | タイムゾーン |
| `accessMode` | `direct` または `proxy` | 後述（推奨は `direct`） |
| `enabled` | `TRUE` | `FALSE` にすると一覧から消える |
| `notes` | 任意 | メモ |

**`users` シート**

| 列 | 値の例 | 意味 |
| --- | --- | --- |
| `email` | `alice@example.com` | Google アカウントのメール |
| `role` | `admin` または `viewer` | `admin` は全データ閲覧 + `/admin` 画面が使える |
| `enabled` | `TRUE` | `FALSE` にするとログイン自体が拒否される |

**`acl` シート**（重要: ここで「誰がどのデータを見られるか」を決める）

| 列 | 値の例 | 意味 |
| --- | --- | --- |
| `email` | `bob@example.com` | viewer ユーザーのメール |
| `sourceId` | `src-001` または `*` | 許可するソース ID。`*` で全許可 |
| `permission` | `read` | 現状 `read` のみ |

`admin` ロールのユーザーはこのシートに書かなくても全部見える。
`viewer` のユーザーは **ここに行を入れない限り何も見えない**。

**`theme` / `events` / `layouts` シート** は最初は空で OK（後で必要に応じて編集）。

> **`layouts` シートについて**: ソースごとにカスタムダッシュボードレイアウトを JSON で定義できます。
> 設定しなければデフォルトレイアウトが使われ、`remote-ftp` スキーマではデータからの自動レイアウト生成もあります。
> 詳しくは [`05_admin_guide.md`](05_admin_guide.md) のカスタムレイアウトの章を参照。

### 1-4. Registry スプレッドシートの ID を控える

ブラウザのアドレスバーを見る:

```
https://docs.google.com/spreadsheets/d/1abcDEF...xyz/edit#gid=0
                                      ┗━━━━━━━━━┛
                                      ここがID
```

`/d/` と `/edit` の間の長い文字列（40 文字前後）を **コピーしてメモ帳に保存** する。
これが **Registry Spreadsheet ID**。後で `VITE_REGISTRY_SPREADSHEET_ID` という設定名で GitHub に登録します。

### 1-5. 実データ用スプレッドシートを準備する（データ源ごと）

実測値は Registry とは別のスプレッドシートに入る。3 通りのパターンがある:

#### パターンA: リモートFTP データ（プロジェクトで新規作成）

1. 1-1 と同じ手順で空のスプレッドシートを 1 枚作り、名前を `SoilSensorFTP` のように付ける
2. アドレスバーから ID をコピー（後で `FTP_SPREADSHEET_ID` として登録）
3. シート（タブ）を以下のように準備する:
   - `sensor_raw`（ヘッダ不要。取り込みスクリプトが自動追記）
   - `sensor_9am`（ヘッダ不要。取り込みスクリプトが自動追記）
   - `alert_rules`（アラート条件定義。ヘッダ + 初期ルールを入力。詳細は下記）
   - `alerts`（アラート検知結果。ヘッダのみ入力。詳細は下記）
4. このスプレッドシートを **後述のサービスアカウントに「編集者」共有** する（手順 5-4 で説明）

##### `alert_rules` シートの初期設定

A1〜F1 にヘッダ、2 行目以降に検知条件を入力:

| enabled | field | operator | value | alert_type | message |
|---------|-------|----------|-------|------------|---------|
| TRUE | bulk_ec | == | 32.768 | sensor_fault | 接触不良・故障の疑い |
| TRUE | vwc | == | 3276.8 | sensor_fault | 接触不良・故障の疑い |
| TRUE | battery2 | < | 2.5 | low_battery | 電池残量低下 |

> 後から行を追加・変更するだけでアラート条件を変更できます。このシートが無くても上記と同等のデフォルトが適用されます。

##### `alerts` シートの初期設定

A1〜H1 にヘッダのみ入力（データ行は自動追加される）:

```
timestamp, detected_at, alert_type, site_id, addr, sensor_number, details, status
```

> アラートメール送信の設定は [`04_gas_setup.md`](04_gas_setup.md) のセクション C を参照。

#### パターンB: M5Stack / Mechatrax の **新規** 案件

1. ユーザー（or プロジェクト）でスプレッドシートを 1 枚作る
2. GAS 受信スクリプトを貼り付けて WebApp 化する（手順は [`04_gas_setup.md`](04_gas_setup.md)）
3. ID をコピーして Registry の `sources` に登録

#### パターンC: M5Stack / Mechatrax の **既に運用中** の案件

ユーザー（案件主）が既に持っているスプレッドシートをそのまま参照する。
以下のどちらか方式を選ぶ:
　
- **方式 direct（推奨・運用がシンプル）**: 案件主に依頼して、ダッシュボードを使う **全 Google アカウントを「閲覧者」共有** してもらう。`accessMode=direct` で登録。
- **方式 proxy（Drive 共有を絞りたいとき）**: 案件主に依頼して、Registry を持つ Google アカウント **1 つだけに「閲覧者」共有** してもらう。`accessMode=proxy` で登録（[`04_gas_setup.md`](04_gas_setup.md) の read_proxy デプロイが必要）。

## 2. Google Cloud プロジェクト作成

### 2-1. プロジェクトを新規作成

1. <https://console.cloud.google.com> を開く
2. 画面上部のプロジェクト選択ドロップダウン（「プロジェクトの選択」と表示されている）をクリック
3. 右上の「新しいプロジェクト」をクリック
4. プロジェクト名を入力（例: `soil-sensor-data-monitoring`）
5. 「作成」をクリック
6. 作成後、上部のドロップダウンで作成したプロジェクトが選択されていることを確認

### 2-2. API を有効化する

1. 左メニュー「APIとサービス」→「ライブラリ」を開く
2. 検索欄に `Google Sheets API` と入力 → 表示された結果をクリック → 「有効にする」をクリック
3. 同様に `Google Drive API` を検索 → 「有効にする」をクリック（バックアップ機能を使う場合のみ）

> **確認方法**: 「APIとサービス」→「有効な API とサービス」に Sheets API が表示されていれば OK。

## 3. OAuth 同意画面の設定（Web ダッシュボード用）

Web ダッシュボードからユーザーが Google ログインするために必要。

### 3-1. 同意画面を構成

1. 左メニュー「APIとサービス」→「OAuth 同意画面」を開く
2. User Type で「**外部**」を選択 →「作成」
3. 以下を入力：
   - **アプリ名**: 任意（例: `Soil Sensor Dashboard`）
   - **ユーザーサポートメール**: 自分のメールアドレスを選択
   - **デベロッパーの連絡先情報**: 自分のメールアドレスを入力
4. 「保存して次へ」をクリック

### 3-2. スコープ

1. 「スコープを追加または削除」をクリック
2. フィルタで `spreadsheets` を検索
3. `Google Sheets API` の `.../auth/spreadsheets.readonly` にチェック
4. 「更新」→「保存して次へ」

### 3-3. テストユーザー

1. 「+ ADD USERS」をクリック
2. 自分のメールアドレスと、ダッシュボードを使う人のメールアドレスを追加
3. 「保存して次へ」→「ダッシュボードに戻る」

> **注意**: 公開ステータスが「テスト」の間は、ここに登録したユーザーしかログインできない。

## 4. OAuth クライアント ID の作成（Web ダッシュボード用）

### 4-1. クライアント ID を作成

1. 左メニュー「APIとサービス」→「認証情報」を開く
2. 上部の「＋ 認証情報を作成」→「OAuth クライアント ID」を選択
3. 以下を入力：
   - **アプリケーションの種類**: `ウェブ アプリケーション`
   - **名前**: 任意（例: `Soil Sensor Web`）
   - **承認済みの JavaScript 生成元**: 「+ URI を追加」をクリックして以下を入力
     - `https://<your-github-username>.github.io`（GitHub Pages のドメイン）
     - 開発時に使う場合は `http://localhost:5173` も追加
   - **承認済みのリダイレクト URI**: 何も追加しない（空のまま）
4. 「作成」をクリック

### 4-2. Client ID を控える

作成後に表示されるダイアログに **クライアント ID** が表示される（`xxxx.apps.googleusercontent.com` の形式）。
これをコピーしておく。

> **保存先**: GitHub リポジトリの Settings → Secrets and variables → Variables → `VITE_GOOGLE_CLIENT_ID` に登録する（後で [02_github_actions.md](02_github_actions.md) で設定）

## 5. サービスアカウントの作成（GitHub Actions 用）

GitHub Actions がスプレッドシートに自動書き込みするために使うアカウント。

### 5-1. サービスアカウントを作成

1. 左メニュー「APIとサービス」→「認証情報」を開く
2. 上部の「＋ 認証情報を作成」→「サービスアカウント」を選択
3. **ステップ 1「サービスアカウントの詳細」**:
   - **サービスアカウント名**: 任意（例: `github-actions`）
   - **サービスアカウント ID**: 自動入力される（変更不要）
   - **説明**: 任意（空でも OK）
   - 「作成して続行」をクリック
4. **ステップ 2「権限（省略可）」**:
   - ⚠️ **何も選ばずに「続行」をクリック**する
   - ロールは不要（スプレッドシートの共有設定で権限を付与するため）
5. **ステップ 3「アクセス権を持つプリンシパル（省略可）」**:
   - 何も入力せず「完了」をクリック

### 5-2. JSON 鍵を作成してダウンロード

1. 「認証情報」画面に戻る → 下部の「サービスアカウント」セクションに作成したアカウントが表示される
2. そのメールアドレス（`github-actions@soil-sensor-data-monitoring.iam.gserviceaccount.com` のような形式）をクリック
3. 上部の「鍵」タブをクリック
4. 「鍵を追加」→「新しい鍵を作成」をクリック
5. キーのタイプで「**JSON**」を選択 →「作成」をクリック
6. JSON ファイルが自動ダウンロードされる

> ⚠️ **セキュリティ注意**:
> - このファイルは **絶対に Git リポジトリに置かない**
> - ダウンロード後、安全な場所に保管するか、GitHub に登録後に削除する

### 5-3. サービスアカウントのメールアドレスを控える

「認証情報」画面のサービスアカント一覧に表示されるメールアドレスをコピーしておく。
形式: `<アカウント名>@<プロジェクトID>.iam.gserviceaccount.com`

### 5-4. スプレッドシートをサービスアカウントに共有

1. 手順 1 で作成した Google スプレッドシートを開く
2. 右上の「共有」ボタンをクリック
3. 「ユーザーやグループを追加」欄に、5-3 で控えたサービスアカウントのメールアドレスを貼り付け
4. 権限を「**編集者**」に設定
5. 「送信」をクリック（「通知を送信」のチェックは外してよい）

> これにより、サービスアカウントがスプレッドシートを読み書きできるようになる。

### 5-5. GitHub Secrets に JSON を登録

1. GitHub でリポジトリを開く → Settings → Secrets and variables → Actions
2. 「New repository secret」をクリック
3. **Name**: `GOOGLE_SERVICE_ACCOUNT_JSON`
4. **Secret**: ダウンロードした JSON ファイルをテキストエディタで開き、**中身を全文コピー**して貼り付け
5. 「Add secret」をクリック

スプレッドシートのバックアップを Google Drive に自動保存したい場合のみ設定する。
不要ならこのセクションはスキップしてよい。

### 6-1. バックアップ用フォルダを作成

1. <https://drive.google.com> を開く
2. 「+ 新規」→「新しいフォルダ」をクリック

### 6-2. フォルダ ID を確認

作成したフォルダを開いたときの URL:

```
https://drive.google.com/drive/folders/ここがフォルダID
```

3. 権限を「**編集者**」に設定 →「送信」

### 6-4. GitHub Secrets に登録

- **Name**: `DRIVE_BACKUP_FOLDER_ID`
- **Secret**: 6-2 で確認したフォルダ ID

## トラブルシューティング

### 「API が有効ではありません」エラー

→ 手順 2-2 で Sheets API を有効化したか確認。有効化後、反映に 1〜2 分かかることがある。

### OAuth ログインで「アクセスがブロックされました」

→ 手順 3-3 でテストユーザーにそのメールアドレスを追加したか確認。

### GitHub Actions が「Permission denied」でスプレッドシートに書き込めない

→ 手順 5-4 でサービスアカウントのメールをスプレッドシートに「編集者」で共有したか確認。
→ メールアドレスにタイプミスがないか確認（`.iam.gserviceaccount.com` で終わるはず）。

### JSON 鍵ファイルを紛失した

→ サービスアカウントの「鍵」タブから古い鍵を削除し、新しい鍵を作成すれば OK。
